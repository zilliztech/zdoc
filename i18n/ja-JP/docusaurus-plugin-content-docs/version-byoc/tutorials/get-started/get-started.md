# Zilliz Cloudで始めよう

この章では、Zilliz Cloudを始めるための資料を提供します。

Milvusは最も人気があり、最も急速に成長しているオープンソースのベクトルデータベースです。Zillizは完全に管理されたMilvusサービスを提供し、セキュリティを考慮してベクトル検索アプリケーションの展開とスケーリングを簡素化し、複雑なインフラストラクチャを構築および維持する必要を排除します。

## 概要について{#overview}

Zilliz Cloudは、中心にMilvusを使用していますが、多くのクラウドネイティブ機能とエンタープライズグレードのサービスを追加しています。次の図は、インフラストラクチャ、コントロールプレーン、データプレーンの3つのレイヤーで構成されるZilliz Cloudアーキテクチャを示しています。

![JSlVwqaoXhdrxObQyuLcMpZOnHg](/img/JSlVwqaoXhdrxObQyuLcMpZOnHg.png)

### アーキテクチャの説明{#architecture-explained}

インフラストラクチャ層は、マルチクラウドKubernetesクラスターとブロック/オブジェクトストレージを通じて、必要なコンピューティングおよびストレージ機能を提供します。これにより、Milvusポッドとコントロールプレーンサービスをさまざまな可用性ゾーンで実行できるようになり、クラウドストレージ内のベクトルおよびインデックスデータの永続化も可能になります。レイテンシ、居住地、またはコンプライアンス要件に最も適した地域でZilliz Cloudクラスターを作成することができます。

インフラストラクチャ層の上に、Zilliz CloudはMilvusがバックアップするクラスターとすべての付加価値サービスを、コントロールプレーンとデータプレーンの2つのプレーンに分散された階層構造に配置します。この階層構造を組織の構造にマッピングして、複数のプロジェクトを作成し、組織内のセクターやブランチ間でデータを分離することができます。  

- **コントロールプレーン**

    コントロールプレーンには、いくつかの付加価値サービスがあります。これらのうちいくつかはグローバルサービスであり、他のものはプロジェクト固有またはクラスター固有です。これらのサービスを組み合わせることで、Milvusクラスターのプロビジョニング、クラスターデータのバックアップと復元、特定のデータソースからのデータの移行とインポート、クラスターのリソース使用量とクエリパフォーマンスの監視が可能になります。

- **データプレーン**

    データプレーンには、クラスタとして知られるさまざまな種類のコンピューティングリソースを持つMilvusインスタンスが格納されています。各クラスタには独自の接続エンドポイントがあります。オープンソースのMilvusインスタンスと同じように、Zilliz Cloudクラスタと通信することができます。

### エンハンスメント{#enhancements}

各Zilliz Cloudクラスターには、コアで実行されているMilvusインスタンスがありますが、Zilliz Cloud上で実行されているMilvusインスタンスは、オープンソースの対応物と同一ではありません。Zillizは、特にクラウドサービスにおいてMilvusを強化しました

- **カーディナル検索エンジン**

    Zilliz Cloudは、ハードウェアリソースの利用に最適化された高度な近似近傍探索(ANNS)アルゴリズムを採用したプロプライエタリなマルチスレッドC++ベクトル検索エンジンであるCardinalを発表しました。Zillizは、自己管理型のMilvusデプロイメントと比較して、クエリスループット(QPS)が10倍高くなっています。Zilliz Cloudのクエリレイテンシは、同様のワークロードに対してバニラのMilvusよりも50ー70%低くなることがよくあります。この強化されたエンジンは、Milvus APIを使用しながら、オープンソースのデフォルトを超えた検索を大幅に加速し、ユーザーフレンドリーなままです。

- **オートインデックス**

    オープンソースのMilvusでは、インデックス(例: IVF、HNSW)の構築には手動での選択とチューニングが必要です。Zilliz CloudのAutoIn dexは、各データセットに最適なインデックスを選択して構築することで、これを自動化します。データとクエリを分析して、最適な戦略(IVF、グラフベース、フラットサーチ)を特定し、推測作業や退屈なチューニングを排除します。これにより、エンジニアリング時間が節約され、パフォーマンスが向上し、AutoIn dexは手動構成に比べてインデックス構築を約40%高速化します。

## スケーラビリティ{#scalability}

Zilliz Cloudは、コンピューティングリソースの追加/削除による垂直方向のスケーラビリティと、レプリカの追加/削除による水平方向のスケーラビリティの両方を提供します。垂直方向のスケーリングはクラスターの容量に影響を与え、水平方向のスケーリングはクエリのスループットと可用性に影響を与えます。 

### 垂直スケーリング{#vertical-scaling}

データが増えるにつれて、データの書き込みに影響を与える制約に直面する可能性があります。このような問題に対処するには、ワークロードまたはストレージ要件の変動に合わせてコンピューティングユニット(CU)の数を調整できます。

![VRKwwXlIVhYlVobOD1Rc9ab2nXb](/img/VRKwwXlIVhYlVobOD1Rc9ab2nXb.png)

CUは、クラスタに割り当てられた物理リソースを測定するために使用される単位です。CPUまたはメモリ使用量の増加に応じてCUをスケールアップすることでクラスタの容量を増やし、需要が低い期間にはコストを削減するためにスケールダウンすることができます。

Zilliz Cloudには、**Performance-optimized**、**Capacity-optimized**、**Extended-capacity**の3種類のCUがあります。クラスタの容量は、クラスタが使用するCUの種類によって異なります。詳細については、[適切なCUを選択してください](./cu-types-explained)を参照してください。

Zilliz Cloudでは、手動と自動の両方の垂直スケーリングが可能です。詳細については、[スケールクラスタ](./scale-cluster)を参照してください。 

### 水平スケーリング{#horizontal-scaling}

各Zilliz CloudクラスターはMilvusインスタンスを実行します。したがって、Zilliz Cloudクラスター内のリソースは、オープンソースのMilvusインスタンスとほぼ同じです。次の図は、Zilliz Cloudクラスターの基本的なアーキテクチャを示しています。

![ZZwTwKFRlhjC4WbFgGUcLsKVn6d](/img/ZZwTwKFRlhjC4WbFgGUcLsKVn6d.png)

小さなデータセットを持つユーザーのQPSボトルネックに対処するために、レプリカを追加することでクエリワークロードを分散させ、最終的にクエリスループットと可用性を向上させることができます。クラスターレプリカは、クラスターの同一のコピーとして機能します。クエリ毎秒(QPS)ボトルネックに遭遇した場合は、クラスターのレプリカを作成してクエリスループットと可用性を向上させることを検討してください。詳細については、[レプリカの管理](./manage-replica)を参照してください。

## セキュリティ{#security}

データを保護するために、Zilliz Cloudは、承認と認証、ネットワークの分離、データの暗号化、バックアップと復元など、さまざまな側面で堅牢なセキュリティ対策を提供します。

- **アクセス制御**

    - 組織、プロジェクト、クラスターレベルでの階層化されたアクセス制御。

    - 適切な権限を持つ事前定義された役割。

詳細については、[アクセス制御](./access-control)を参照してください。

- **監査ログ**

    - コントロールプレーンのアクティビティとデータプレーンの操作の記録を保持します。

    - 生成されたデータプレーンログをブロックまたはオブジェクトストレージにストリーミングします。

    - サードパーティのログ分析ツールを通じてデータの洞察を提供します。

    詳細については、[監査する](./auditing)を参照してください。

- **認証システム**

    - **OAuth 2**プロトコルを使用してユーザーを認証します。

    - APIキーとクラスター資格情報の両方を使用してクラスターにアクセスできます。

    - シングルサインオン(SSO)を介してユーザーログインを有効にします。

    詳細については、[認証プロセス](./authentication)を参照してください。

- **データの整合性**

    - 手動および自動バックアップと復元が可能です

    - 適切な保持期間を持つごみ箱を実装します。

    詳細については、[バックアップと復元](./backup-and-restore)と[使用ごみ箱](./use-recycle-bin)を参照してください。

- **輸送中の暗号化**

    クライアントは、ZillizクラスターへのHTTPSまたはgRPC接続を確立します。HTTPS/gRPC接続では、TLS 1.2（またはそれ以上）プロトコルとAES-256（256ビット高度暗号化標準）を使用して、転送中のユーザーデータを暗号化します。

- **保存時の暗号化（server-side-encryption）**

    Zilliz Cloudは、AES-256(256-bit Advanced Encryption Standard)暗号化アルゴリズムを使用してAWS S 3に保存されたデータを暗号化します。

- **ネットワークポリシー**

    - アクセス元を制限するために、ホワイトリスト上のカスタムCIDRブロックを許可します。

    - リストされていないIPアドレスからのアクセスをフィルタリングします。

    詳細については、[ホワイトリストの設定](./setup-whitelist)を参照してください。

- **非公開リンク**

    - クラウドプロバイダーに合わせたプライベートリンクを介してクラスターへの接続を容易にします。

    - VPCからZilliz Cloudコントロールプレーンへのプライベート接続を可能にします。

    詳細については、[プライベートエンドポイントを設定する](./setup-a-private-link)を参照してください。
